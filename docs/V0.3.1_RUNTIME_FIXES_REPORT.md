# Arab Poets Platform — v0.3.1 Runtime Fixes Report

**Date:** 20 July 2026  
**Baseline:** v0.3.0  
**Scope:** Issues reproduced during local CMS, seed, Cloudinary, localization, Versions, and public-site testing.

## Executive result

v0.3.1 addresses the runtime failures found after applying the v0.3 migration and seed. The release removes the failing Next.js image-proxy path for Cloudinary media, repairs seed-media idempotency, regenerates the Payload Lexical import map, initializes the Admin UI in Arabic, enables publication status per locale, reduces unnecessary Versions, and cuts the public homepage CMS reads from approximately six to three concurrent requests.

The codebase passes generated types, generated import map, ESLint, TypeScript, and the production Webpack build. Applying the new migration and running the seed against the project Supabase database must still be completed on the development machine because this execution environment does not have the project's database or Cloudinary credentials.

## Fixed issues

### 1. Cloudinary and Next Image timeouts

- Added a custom Next.js image loader at `src/lib/image-loader.ts`.
- Cloudinary assets now load directly from the Cloudinary CDN with bounded width, automatic format, and automatic or requested quality.
- Requests no longer pass through `/_next/image`, removing the server-side Cloudinary fetch that timed out locally.
- Legacy `prefix=media` query parameters are removed before rendering.
- Cloud storage now uses `disablePayloadAccessControl: true`, preventing Payload's storage hook from adding the prefix query to new URLs.
- Media mapping prefers `cloudinarySecureUrl` over the mutable Payload `url` field.

### 2. Seed media verification and duplicate prevention

- Added the hidden, indexed, unique `media.internalSeedKey` field.
- Seed media is identified by stable keys such as `seed-media:istanbul-event.png`, not Cloudinary's randomized filename.
- Existing v0.3 media is adopted once using its Arabic alt text, so the first v0.3.1 seed does not need to upload another copy.
- Older duplicate media records are deliberately left untouched for safe manual review.
- `seed:verify` now checks all five fixture media records and rejects URLs containing `prefix=`.

### 3. Payload Lexical import map

- Regenerated `src/app/(payload)/admin/importMap.js`.
- The map now contains `RscEntryLexicalField`, `RscEntryLexicalCell`, `LexicalDiffComponent`, and the required Lexical client features.
- Added `predev` and `prebuild` generation so stale import maps are less likely to recur.

### 4. Arabic Payload Admin interface

- Arabic and English remain configured through `@payloadcms/translations`.
- Added a one-time `/admin` language migration in `src/proxy.ts`.
- The first Admin request on v0.3.1 writes `payload-lng=ar` and redirects once.
- After initialization, Payload's normal language control remains authoritative, so a later English selection is preserved.
- The content-locale selector and Admin-interface language remain separate controls.

### 5. Independent Arabic and English publication status

Enabled Payload's localized-status workflow for:

- Events
- Event Series
- People
- Literary Works
- Homepage

The new migration uses Payload's official `localizeStatus` migration utility to reconstruct locale statuses from version history before moving `_status` into locale tables. Existing shared publication state is initially carried to the localized model; future Arabic and English publishing can then be managed independently.

Public queries now request only the active locale. Consequently, a document published in Arabic but still Draft in English is available under `/ar` and absent under `/en`.

The sitemap was also updated to include event and person URLs only in locales where those documents are publicly visible.

### 6. Version-history noise

- Event and Person autosave interval changed from 3 seconds to 15 seconds.
- Version retention reduced from 80 to 30 per document.
- Event Series, Literary Works, and Homepage also retain at most 30 versions.
- A normal repeated seed no longer updates existing versioned fixture records.
- Fixture refresh remains possible through `SEED_UPDATE_EXISTING=true`.

Payload will still create legitimate versions and snapshots during editing and locale-specific publishing; those are part of the recovery and publication workflow rather than duplicate documents.

### 7. CMS query performance

- List and detail loaders now fetch only the requested locale instead of both languages on every request.
- Homepage CMS reads were reduced from approximately six to three concurrent operations: Events, People, and Homepage.
- Event and Person detail pages retain direct slug queries.
- Payload query depth remains bounded.

This removes avoidable database work. Final response-time measurement must be repeated on the development machine against its Supabase connection.

### 8. Favicon

- Added a 128×128 optimized `src/app/icon.png`.
- This removes the observed `/favicon.ico`-equivalent metadata icon failure in App Router clients.

## Database migration

New migration:

```text
20260720_122756_cms_runtime_fixes
```

It performs the following:

1. Migrates status history to Arabic/English locale tables for all configured versioned entities.
2. Preserves current publication state using Payload's version-history migration logic.
3. Aligns localized status columns with the existing PostgreSQL enum types and defaults.
4. Recreates status indexes on locale tables.
5. Adds the unique `media.internal_seed_key` column and index.

The rollback collapses publication status back to the default Arabic locale before restoring the old schema.

## Required upgrade sequence

Run from the extracted v0.3.1 project using the existing `.env.local`:

```bash
npm ci
npm run migrate:status
npm run migrate
npm run seed
npm run seed
npm run seed:verify
npm run verify
npm run dev
```

The first v0.3.1 seed adopts the old fixture media and writes stable seed keys. The second normal seed should not create records or new content Versions.

## Manual acceptance checks

1. Open `/admin`; the Admin interface should initialize in Arabic after one redirect.
2. Open an Event Version; no `RscEntryLexicalField` import-map warning should appear.
3. Confirm Arabic and English status can be changed independently.
4. Publish Arabic while leaving English Draft; verify the Event appears only under `/ar`.
5. Inspect Network requests for Cloudinary media; image URLs should target `res.cloudinary.com` directly and no request should use `/_next/image` for those assets.
6. Confirm no media URL contains `prefix=media`.
7. Run `seed:verify`; all checks should pass.
8. Reload `/ar` several times and record server response times after the initial cold request.
9. Confirm `/icon.png` returns 200.

## Remaining non-blocking items

- Email delivery is still console-only until an email adapter is configured.
- Existing duplicate Media rows from earlier v0.3 seed runs are not automatically deleted.
- Full Lexical formatting is still flattened in the public website content mapper; this should be addressed when rich poetry formatting becomes part of the public content scope.
- The in-memory form limiter should be replaced with a shared limiter before multi-instance production deployment.
