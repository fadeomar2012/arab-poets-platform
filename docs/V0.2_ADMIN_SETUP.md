# v0.2 — Payload + Supabase + Cloudinary

## 1. Install dependencies

```bash
npm install --legacy-peer-deps
```

The archive intentionally does not include `node_modules` or a stale lockfile. Commit the new `package-lock.json` created on your computer.

## 2. Environment

Copy `.env.example` to `.env.local` and paste your existing values. Never commit `.env.local`.

- `DATABASE_URL`: Supabase runtime connection. Transaction pooler is appropriate for transient/serverless runtime connections.
- `DATABASE_MIGRATION_URL`: direct connection or session pooler used by migration commands.
- `PAYLOAD_SECRET`: long random server-only secret.
- `NEXT_PUBLIC_SERVER_URL`: `http://localhost:3000` locally.
- Cloudinary secrets remain server-only.

## 3. Create and apply the initial schema

The recommended Supabase workflow is migration-first:

```bash
npm run migrate:create -- initial
npm run migrate
```

The migration wrapper loads `.env.local`, then temporarily substitutes `DATABASE_MIGRATION_URL` for `DATABASE_URL`.

For isolated development only, `PAYLOAD_DB_PUSH=true` enables Payload/Drizzle push mode. Keep it `false` for staging and production.

## 4. Start and create the first administrator

```bash
npm run dev
```

Open:

```text
http://localhost:3000/admin
```

The first user created is forced to the `admin` role. Later users default to `editor`.

## 5. Seed the basic taxonomies

```bash
npm run seed
```

This adds starter event types, countries, and Istanbul.

## 6. Test Cloudinary

In **Admin → Media → Create New**, upload a JPEG, PNG, WebP, or AVIF.

Expected result:

- File stored in `arab-poets/media` in Cloudinary.
- Cloudinary asset metadata stored in Supabase.
- Removing the Media document also calls Cloudinary destroy with CDN invalidation.

## 7. Test real content

Create in this order:

1. Event Type
2. Country
3. City
4. Person
5. Event

Save the Event as a draft, preview it, then publish it. Published CMS records replace the bundled preview data. If Payload cannot connect, public pages fall back to bundled mock data so the website remains reviewable.

## 8. Test public forms

Submit:

- `/ar/contact`
- `/ar/participate`

Then verify the records under **Admin → Inboxes**. A honeypot is included, and direct anonymous writes to the underlying Payload inbox collections are blocked. Distributed production rate limiting and email notifications remain in the next sprint.

## 9. Netlify

Before deployment:

1. Run and commit the migration files.
2. Apply migrations to Supabase.
3. Add all environment variables to Netlify.
4. Set both public URLs to the final `https://...netlify.app` URL.
5. Keep `PAYLOAD_DB_PUSH=false`.
6. Deploy, then open `/admin`.
