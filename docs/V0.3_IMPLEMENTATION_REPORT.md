# Arab Poets Platform — v0.3 Implementation Report

**Date:** 20 July 2026  
**Scope:** CMS hardening, bilingual admin, website integration, seed data, forms, media, migrations, and production verification.

## Executive result

The project is now a testable Payload CMS baseline rather than a presentation-only scaffold. The public event and people pages read Payload data directly, CMS failures are no longer silently hidden in staging/production, the Payload admin interface supports Arabic and English, and the project includes an idempotent bilingual seed plus a verification command.

## Implemented changes

### Payload and administration

- Enabled Arabic and English Payload Admin UI through `@payloadcms/translations`.
- Added bilingual labels for collections, groups, fields, options, and globals.
- Preserved Arabic as the default content locale with English fallback.
- Added URL validation for websites, social links, media links, and partner links.
- Added automatic slug generation with Arabic transliteration support.
- Added server-side validation that a selected city belongs to the selected country.
- Replaced free-text event timezone values with a controlled IANA timezone list.
- Made media alternative text required.
- Kept Admin/Editor access boundaries and public read restrictions.

### Website and CMS integration

- Added direct Payload queries by event/person slug instead of loading every record.
- Added strict CMS behavior for staging/production; mock content is opt-in only.
- Made CMS-backed routes dynamically rendered so newly published content does not require a code change or redeploy during testing.
- Added `ongoing` event status using both start and end dates.
- Propagated CMS media URL, alt text, caption, and credit into public components.
- Added real profile-image support for people.
- Connected the Homepage global to featured-event/institutional hero behavior.
- Made sitemap generation CMS-aware and dynamic.
- Improved rich-text plain rendering to preserve paragraph and line breaks.

### Public forms

- Fixed Payload create calls and explicit draft behavior.
- Kept direct anonymous collection writes blocked while allowing trusted Route Handlers to create validated records.
- Added Zod validation, honeypot checks, structured errors, and best-effort rate limiting.
- Added phone and portfolio URL support to participation requests.
- Added user-facing handling for HTTP 429 responses.

### Cloudinary and environment handling

- Cloudinary remains server-side only.
- Local Payload media storage now works automatically when Cloudinary credentials are absent.
- Added `.env.example` with runtime/migration database separation and CMS strict-mode variables.
- Expanded CORS/CSRF origins for the configured site and Netlify deployment/preview URLs.
- No `.env` or secret file is included in the distributable ZIP.

### Seed and verification

- Replaced the limited seed with an idempotent bilingual seed.
- Seed content includes:
  - three event types;
  - four countries and four matching cities;
  - one event series;
  - four public people and one hidden draft person;
  - three literary works;
  - two published events and one draft event;
  - two partners;
  - media records and images;
  - Site Settings and Homepage globals.
- Corrected environment loading so `.env.local`/`.env` are loaded before importing Payload config.
- Added `npm run seed:verify` to check uniqueness, publication status, public visibility, media, and homepage setup.

### Database and deployment

- Added migration `20260720_110905_cms_hardening`.
- Migration changes:
  - requires `media_locales.alt`;
  - creates controlled event timezone enums;
  - changes the default timezone to `Asia/Gaza`;
  - sets the default attendance mode to `open`;
  - mirrors changes in event version tables.
- Fixed the production build command to the verified Webpack path: `next build --webpack`.
- Added Netlify Node 22, memory, telemetry, and basic security-header settings.

## Required test sequence on the development machine

```bash
npm ci
npm run migrate:status
npm run migrate
npm run seed
npm run seed
npm run seed:verify
npm run verify
npm run dev
```

Expected behavior:

1. The second seed run creates no duplicate slugs or filenames.
2. `/admin` allows choosing Arabic or English UI.
3. `istanbul-poetry-forum-2026` and `arab-poetry-festival-eighth` are public.
4. `cairo-contemporary-poetry-evening` remains a draft and is not public.
5. The hidden draft person does not appear in the public directory.
6. Publishing or editing an event is reflected on the dynamic public route.
7. Contact and participation forms create inbox records through their public API routes.
8. Cloudinary images load when credentials are configured; local storage works when they are not.

## Known limitations and next-stage work

- Rate limiting is in-memory and best-effort. For multi-instance production deployment, replace it with a shared Redis/Upstash limiter.
- Homepage Globals are partially connected. Additional fixed landing-page sections can be moved into CMS fields in a later content-modeling phase.
- Rich text currently preserves readable text and line breaks but does not render the full Lexical formatting model. A safe Lexical renderer is recommended for poetry, lists, links, and quotations.
- Database migration and seed execution must be confirmed from a network that can reach the configured Supabase project.
- The generic Webpack warning should be captured in Netlify logs if it appears there; it did not fail the build or expose an actionable module warning locally.
