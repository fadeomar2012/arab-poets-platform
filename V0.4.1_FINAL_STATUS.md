## G. Cold and warm performance measurements

Measured against `npm run build && npm run start` (production).

- **Static public routes** (`/ar`, `/en`, `/about`, `/events`, `/people`,
  `/gallery`) — the routes originally reported as 10–25s: **3–82 ms warm**
  (prerendered SSG). The original slowness was Turbopack **dev** compilation, not
  production.
- **Dynamic detail routes** (`/events/[slug]`, `/people/[slug]`): dominated by
  Supabase round-trips. A raw `pg` benchmark during this session measured
  **connect ≈ 3540 ms, per-query ≈ 320 ms** — abnormally high (the same page
  code measured ~0.6–0.75 s earlier in the session when the managed DB was
  responsive). Payload's `depth:1` relation population issues several queries per
  render, so page time ≈ (query count) × (current per-query latency). This is
  external managed-DB latency (evidence: raw pg timing with zero app code), not a
  regression. Mitigations applied: connection keep-alive, larger warm pool.
- **Dev vs prod**: dev cold compile per route is multi-second (Turbopack, one
  time); production warm static requests are single-digit ms.

## I. Known risks

- **Supabase free-tier latency** makes dynamic detail pages slow under load /
  after heavy use. Production should use an appropriately provisioned DB and may
  benefit from prerendering detail pages (`generateStaticParams`) — deferred
  because it interacts with the on-page Draft Mode read.
- **Draft-content rendering** was verified for security/flow (secret, redirect,
  cookie, isolation). Rendering an actual unpublished record end-to-end was not
  automated (requires creating draft content via an authenticated admin session).
- **axe color-contrast** can flag mid-animation frames; tests disable motion and
  wait for the reveal to settle. Production users briefly see the fade-in.
- **Admin dashboard** interactions are not covered by automated tests (needs
  login credentials).
